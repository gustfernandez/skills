# pr-finalize

## Overview

Run this after you have committed and pushed the code fixes that `executor` applied from a `pr-comments` plan. This skill handles the GitHub-side cleanup:

- Posts `"Fixed in <sha>: <summary>"` as an inline reply on each fix thread.
- Resolves each fix thread.
- Handles the gated Copilot re-request (auto for rounds 1–2; asks for round 3+).

It does **not** touch code, does not commit, and does not push. Those gates remain with the user.

## Inputs

- **Plan slug**: the filename stem of the plan + sidecar pair, e.g. `pr-42-bot-comments-2026-04-30`. The skill reads `~/.claude/plans/<slug>.actions.json`.

## Workflow

### Step 1 — Read sidecar

```bash
cat ~/.claude/plans/<slug>.actions.json
```

Parse: `pr.owner`, `pr.repo`, `pr.number`, `worktree`, `branch`, `round_number`, `fix_threads`.

If the file does not exist or is not valid JSON, abort immediately:
```
Error: ~/.claude/plans/<slug>.actions.json not found or invalid.
Was the plan generated by /pr-comments? Check the slug.
```

### Step 2 — Validate push

Get the current local HEAD and check it matches the upstream:

```bash
LOCAL=$(git -C <worktree> rev-parse HEAD)
UPSTREAM=$(git -C <worktree> rev-parse --abbrev-ref --symbolic-full-name @{upstream} 2>/dev/null)
REMOTE=$(git -C <worktree> rev-parse "${UPSTREAM}" 2>/dev/null)
```

If `LOCAL != REMOTE` or upstream doesn't exist:
```
Error: branch <branch> not pushed yet (or local HEAD is ahead of remote).
Push first, then re-run /pr-finalize <slug>.
```

Get the short SHA for replies:
```bash
SHORT_SHA=$(git -C <worktree> rev-parse --short HEAD)
```

### Step 3 — Resolve fix threads

For each entry in `fix_threads`:

1. Post reply:
   ```bash
   gh api repos/<owner>/<repo>/pulls/<N>/comments/<comment_id>/replies \
     -X POST -f body="Fixed in ${SHORT_SHA}: <summary>"
   ```

2. Resolve thread:
   ```bash
   gh api graphql -f query='
     mutation($tid: ID!) {
       resolveReviewThread(input: {threadId: $tid}) {
         thread { id isResolved }
       }
     }' -F tid=<thread_id>
   ```

On HTTP 4xx/5xx for either call: log `FAILED: <thread_id> — <http status>`, increment a failure counter, and continue with the next thread. Do NOT abort on individual failures. Surface failures in the final summary.

### Step 4 — Gated Copilot re-request

Use `round_number` from the sidecar (the count of Copilot reviews **before** this push round).

**`round_number <= 1`** (this is round 1 or 2 — auto re-request):
```bash
gh pr edit <N> -R <owner>/<repo> --add-reviewer "@copilot"
```
Print: `Re-requested Copilot — round <round_number+1>, auto.`

**`round_number >= 2`** (round 3+ — do NOT re-request; ask instead):
```
Round <round_number+1> — recommend stopping. Copilot has already reviewed <round_number> times; marginal value diminishes quickly.
If you want one more round, run:
  gh pr edit <N> -R <owner>/<repo> --add-reviewer "@copilot"
```
Do not run the command. Let the user decide.

### Step 5 — Summary

Print one line:
```
<repo>#<N>: <k>/<total> fix-threads resolved, <f> failed, commit <SHORT_SHA>, round <round_number+1>, re-requested <yes|asked>.
```

If `f > 0`, list the failed thread IDs after the summary line so the user can investigate and re-run manually.

## Quick reference

| Condition | Re-request behavior |
|---|---|
| `round_number == 0` | Auto — round 1, first Copilot review pass |
| `round_number == 1` | Auto — round 2, reasonable iteration |
| `round_number == 2` | Ask — round 3, time to consider stopping |
| `round_number >= 3` | Ask — round 4+, strong recommendation to stop |

## Common mistakes

- **Running before push.** The SHA-check in Step 2 will catch this. Do not skip it.
- **Using databaseId for resolveReviewThread.** Must use the GraphQL node ID (`PRRT_...`) from the sidecar. The sidecar's `thread_id` field is already the node ID.
- **Expanding the `body` with double quotes without controlling interpolation.** The `SHORT_SHA` variable needs to expand; the summary should not. Use `"Fixed in ${SHORT_SHA}: <literal summary>"` — do not add more `$` variables inside the body.
- **Re-requesting Copilot on round 3+.** The gating logic is `round_number >= 2` → ask. Print the command for the user; do not call it.
- **Silently dropping failed threads.** Log each failure and surface in the final summary. The user needs to know which threads are still unresolved.
